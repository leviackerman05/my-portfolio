Shipping frontend code from a monorepo should feel boring. Push to a branch, preview deploy goes live, merge to main, production updates without manual steps, no "works on my machine" env drift, and a rollback path when something slips through. GitHub Actions makes this achievable for frontend monorepos without a dedicated platform team, but the details matter: cache strategy, preview URLs per PR, environment secrets, and a rollback that takes minutes not hours.

In this guide, we'll build a production-grade CI/CD pipeline for a frontend monorepo using GitHub Actions, covering deploy previews, environment configuration, and rollback strategies that actually get used.

## Monorepo Structure Assumptions

We'll use a typical frontend monorepo layout:

```
frontend-monorepo/
├── apps/
│   ├── admin/          # Enterprise admin (React)
│   ├── marketing/      # Next.js marketing site
│   └── portfolio/      # Personal site (Vite + React)
├── packages/
│   ├── ui/             # Shared component library
│   ├── config-eslint/
│   └── config-typescript/
├── package.json
├── pnpm-workspace.yaml
└── turbo.json
```

Key tooling:

- **pnpm** for workspace-aware installs
- **Turborepo** for task orchestration and remote caching
- **GitHub Actions** for CI/CD
- **Vercel / Netlify / S3+CloudFront** for hosting (patterns apply to all)

## Core CI Workflow

Every push and pull request should run the same quality gates:

```yaml
# .github/workflows/ci.yml
name: CI

on:
  pull_request:
  push:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm turbo lint

      - name: Typecheck
        run: pnpm turbo typecheck

      - name: Test
        run: pnpm turbo test -- --ci

      - name: Build
        run: pnpm turbo build
```

### Why Turborepo in CI

Turborepo skips unchanged packages. In a monorepo with five apps and twelve packages, a one-line fix in `packages/ui` only rebuilds dependents, not the entire tree.

```json
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "dist/**"]
    },
    "lint": {},
    "typecheck": {},
    "test": {
      "dependsOn": ["^build"]
    }
  }
}
```

Enable Turborepo remote caching in CI for faster runs across PRs:

```yaml
      - name: Build
        run: pnpm turbo build
        env:
          TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
          TURBO_TEAM: ${{ vars.TURBO_TEAM }}
```

## Deploy Previews on Pull Requests

Preview deploys are the highest-ROI CI feature for frontend teams. Reviewers click a URL and see the actual UI, not a screenshot.

### Vercel (Simplest for Next.js / Vite)

Vercel's GitHub integration handles previews automatically. For custom control:

```yaml
# .github/workflows/preview.yml
name: Preview Deploy

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  deploy-preview:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - run: pnpm install --frozen-lockfile
      - run: pnpm turbo build --filter=admin

      - name: Deploy to Vercel Preview
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          working-directory: apps/admin
```

### Comment Preview URL on the PR

```yaml
      - name: Comment preview URL
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `🚀 Preview deployed: ${{ steps.deploy.outputs.preview-url }}`
            })
```

### Preview Environment Variables

Previews should use staging credentials, never production secrets:

```yaml
      - name: Build
        run: pnpm turbo build --filter=admin
        env:
          VITE_API_URL: ${{ vars.STAGING_API_URL }}
          VITE_GRAPHQL_ENDPOINT: ${{ vars.STAGING_GRAPHQL_URL }}
```

In GitHub: set `STAGING_*` as **Repository Variables** (non-secret, visible) and keep real secrets in **Environment Secrets** scoped to the `production` environment.

## Environment Configuration

Env config is where CI/CD pipelines silently fail. A few rules:

### 1. Never Commit `.env` Files

Use `.env.example` with placeholder values:

```bash
# apps/admin/.env.example
VITE_API_URL=https://api.staging.example.com
VITE_GRAPHQL_ENDPOINT=https://api.staging.example.com/graphql
VITE_FEATURE_FLAGS_URL=
```

### 2. Use GitHub Environments

Create `staging` and `production` environments in GitHub repo settings. Attach protection rules to `production`:

- Required reviewers
- Deployment branches limited to `main`
- Environment-specific secrets

```yaml
jobs:
  deploy-production:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - name: Build
        run: pnpm turbo build --filter=admin
        env:
          VITE_API_URL: ${{ secrets.PROD_API_URL }}
          VITE_GRAPHQL_ENDPOINT: ${{ secrets.PROD_GRAPHQL_URL }}
```

### 3. Inject at Build Time for SPAs

Vite and Create React App bake env vars at build time. The CI workflow must pass them during `build`, not at deploy. Document which vars are required per app in each app's README.

### 4. Validate Env Vars in CI

Fail fast when required vars are missing:

```typescript
// apps/admin/src/config.ts
const required = ['VITE_API_URL', 'VITE_GRAPHQL_ENDPOINT'] as const;

required.forEach((key) => {
  if (!import.meta.env[key]) {
    throw new Error(`Missing required env var: ${key}`);
  }
});

export const config = {
  apiUrl: import.meta.env.VITE_API_URL,
  graphqlEndpoint: import.meta.env.VITE_GRAPHQL_ENDPOINT,
};
```

## Production Deploy Workflow

Keep production deploys manual or gated. Automatic deploy on every main push is fine only when test coverage and preview review are solid.

```yaml
# .github/workflows/deploy-production.yml
name: Deploy Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - run: pnpm install --frozen-lockfile
      - run: pnpm turbo build --filter=admin
        env:
          VITE_API_URL: ${{ secrets.PROD_API_URL }}
          VITE_GRAPHQL_ENDPOINT: ${{ secrets.PROD_GRAPHQL_URL }}

      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: admin-dist
          path: apps/admin/dist
          retention-days: 30

      - name: Deploy to S3
        run: aws s3 sync apps/admin/dist s3://${{ secrets.S3_BUCKET }} --delete

      - name: Invalidate CloudFront cache
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CF_DISTRIBUTION_ID }} \
            --paths "/*"
```

**Upload artifacts before deploy.** Artifacts are your rollback source of truth.

## Rollback Strategies

Rollback is not "revert the commit and pray." It is redeploying a known-good artifact.

### Strategy 1: Redeploy Previous Artifact (Fastest)

Store build artifacts with the git SHA in the name:

```yaml
      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: admin-dist-${{ github.sha }}
          path: apps/admin/dist
          retention-days: 90
```

Rollback workflow:

```yaml
# .github/workflows/rollback.yml
name: Rollback Production

on:
  workflow_dispatch:
    inputs:
      artifact_sha:
        description: 'Git SHA of the build to redeploy'
        required: true

jobs:
  rollback:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - name: Download artifact
        uses: actions/download-artifact@v4
        with:
          name: admin-dist-${{ inputs.artifact_sha }}
          path: apps/admin/dist

      - name: Deploy to S3
        run: aws s3 sync apps/admin/dist s3://${{ secrets.S3_BUCKET }} --delete

      - name: Invalidate CloudFront cache
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CF_DISTRIBUTION_ID }} \
            --paths "/*"
```

Trigger from GitHub Actions UI. No code change required. Rollback in minutes.

### Strategy 2: Git Revert + Redeploy

For source-level rollbacks:

```bash
git revert <bad-commit-sha>
git push origin main
```

CI runs, builds, deploys. Slower than artifact rollback but fixes the codebase too.

### Strategy 3: Feature Flags for Risky Releases

For large features, wrap in a flag and disable at runtime without redeploying:

```typescript
const showNewAdminWorkflow = featureFlags.isEnabled('new-admin-workflow');
```

Flags do not replace rollback. They buy time.

## Caching for Faster Pipelines

Slow CI erodes trust. Optimize in this order:

1. **pnpm store cache**: `actions/setup-node` with `cache: pnpm`
2. **Turborepo remote cache**: share build outputs across CI runs
3. **Docker layer cache**: if using containerized builds
4. **Path filters**: skip CI when only docs change:

```yaml
on:
  pull_request:
    paths-ignore:
      - '**.md'
      - 'docs/**'
```

## Best Practices

1. **One workflow per concern**: CI, preview, production deploy, rollback stay separate.
2. **Concurrency groups**: cancel stale runs on the same branch.
3. **Frozen lockfile**: always `pnpm install --frozen-lockfile` in CI.
4. **Environment-scoped secrets**: staging and production never share credentials.
5. **Artifact retention**: keep 30–90 days of production builds for rollback.
6. **PR preview links in comments**: make review frictionless.
7. **Post-deploy smoke test**: hit a health endpoint or run a Playwright check against the deploy URL.

```yaml
      - name: Smoke test
        run: curl -f https://admin.example.com/health
```

## Conclusion

CI/CD for frontend monorepos is not about adopting every GitHub Actions feature. It is about predictable paths: every PR gets a preview, every merge to main goes through the same gates, production secrets stay scoped, and rollback means redeploying a stored artifact instead of debugging at 2 AM.

Start with the CI workflow and preview deploys. Add environment separation next. Artifact-based rollback last, but add it before you need it, not after.

If you are wiring this up today, create the `rollback.yml` workflow first. Knowing you can undo a deploy changes how confidently you ship.

Happy deploying!
