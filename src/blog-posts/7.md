Every product team is under pressure to "add AI" in 2025 and 2026. Copilot for codegen, LLM summaries in admin panels, chatbots on support pages, auto-generated configs in CI. Most of these demos work in a slide deck. Fewer of them belong in production next to auth, billing, and permission systems.

After building AI-assisted tools like [Vizion](https://github.com/leviackerman05/vizion) and working on enterprise admin surfaces where access control and audit trails are non-negotiable, a clear line emerged: AI is excellent for **drafting, suggesting, and transforming content**. It is a liability for **authorizing, charging, and granting access** without human-readable, testable, deterministic code in the middle.

This article is about where that line sits, and why some features should stay rule-based no matter how good the model gets.

## The Core Problem: Non-Determinism Meets Accountability

Production systems need:

- **Repeatability**: same input, same output, every time
- **Auditability**: who did what, when, with what authority
- **Reversibility**: rollback, refund, revoke access
- **Testability**: unit tests that pass in CI, not "looks fine to me"

LLMs fail all four by design. Temperature, model version drift, prompt injection, and hallucination mean you cannot sign an SLA on model output the way you sign one on a REST endpoint.

That does not mean "no AI in prod." It means AI should not be the **authority** in prod.

## Auth Flows: Never Let the Model Decide Who You Are

Authentication answers: *Is this person who they claim to be?*

AI should not:

- Parse login forms and "decide" if credentials look valid
- Generate session tokens or JWT claims from natural language
- Interpret OAuth callbacks without strict schema validation
- Replace MFA verification with "the model thinks this is legitimate"

**Why**: a false accept is a security incident. A false reject is annoying. Models optimize for plausible text, not cryptographic certainty.

**What to do instead**:

```typescript
// Deterministic auth stays in code you test
const session = await verifyCredentials(email, passwordHash);
if (!session) throw new UnauthorizedError();

// AI can help AFTER auth, in non-authoritative ways
const welcomeMessage = await generateWelcomeCopy(user.displayName);
```

AI can personalize copy, suggest password strength tips, or summarize account activity **after** identity is established by proven mechanisms (bcrypt, OIDC, WebAuthn, Supabase Auth, etc.).

In Vizion, auth is handled by Supabase `sign_in_with_password` and JWT verification in `verify_token`. The LLM never touches tokens. That boundary should be explicit in every AI product.

## Billing: Money Requires Ledgers, Not Probabilities

Billing answers: *What does this user owe, and did we collect it?*

Never put AI in the path of:

- Calculating invoice totals
- Applying discounts or credits
- Deciding subscription tier changes
- Parsing payment webhooks from Stripe or Razorpay
- Generating refund amounts

**Why**: a hallucinated discount code or misread webhook loses real money. Finance teams need integer cents in a database, not a model's best guess.

**Real incident pattern**:

1. Support pastes a customer complaint into an internal AI tool
2. Model "helpfully" suggests a 90% refund
3. Junior engineer copies suggestion into admin without review
4. Finance discovers the problem at month-end close

**What to do instead**:

- Webhooks parsed by typed handlers with idempotency keys
- Pricing logic in version-controlled code with exhaustive tests
- AI drafts support replies that **link** to refund policy, not execute refunds

```python
# Good: AI drafts, human approves, system executes
draft = llm.generate_support_reply(ticket_context)
# Human reviews draft in Zendesk/Linear
# Refund triggered only by explicit admin action with audit log
admin.refund(order_id, amount_cents=500, reason="billing_dispute", actor=admin_id)
```

## Permission Changes: RBAC Does Not Belong in a Prompt

Enterprise admin tools (the kind I have worked on at scale) live and die on permissions:

- Who can view this account?
- Who can edit roles?
- Who can export PII?
- Who can delete a user?

AI must not:

- Interpret `"give John admin access"` from chat and apply it
- Infer permissions from vague natural language requests
- Bulk-update ACLs based on model-generated JSON without schema validation
- Bypass approval workflows because the model said it was fine

**Why**: permission bugs are data breaches. SOC2 and GDPR auditors ask for logs showing **which human or service account** changed access, not which prompt version was live that day.

**What to do instead**:

```typescript
interface RoleChangeRequest {
  targetUserId: string;
  newRoleId: RoleId; // enum, not free text
  requestedBy: string;
  approvedBy?: string;
}

// Explicit state machine
await permissionService.requestChange(payload);
// Requires manager approval for sensitive roles
// Writes immutable audit event
await auditLog.write({
  action: 'ROLE_CHANGE',
  actor: approvedBy,
  target: targetUserId,
  before: oldRole,
  after: newRoleId,
  timestamp: new Date().toISOString(),
});
```

AI can help admins **find** the right user, **summarize** current permissions, or **draft** a change request form. The apply button runs deterministic code.

## Anything That Needs Audit Trails

An audit trail answers: *What happened, who caused it, and can we prove it in court or to a regulator?*

If a feature requires audit trails, AI should not be the executor. Period.

Examples:

- Granting API keys
- Changing data retention settings
- Exporting customer data
- Merging accounts
- Disabling security controls
- Modifying production feature flags that affect billing or access

**Audit log requirements**:

- **`actor_id`**: human or service account, never "the model"
- **`action`**: enum from a fixed set
- **`resource_id`**: what was affected
- **`before` / `after`**: serializable state diff
- **`timestamp`**: UTC, server-generated
- **`correlation_id`**: tie to request trace

LLM calls can be logged for debugging (`prompt_hash`, `model_version`, `latency_ms`), but they are not substitutes for business event logs.

## Where AI *Does* Belong in Production

Based on Vizion and enterprise admin experience, these patterns work:

### 1. Draft and Suggest (Human in the Loop)

- Support reply drafts
- PR description generation
- Test case suggestions
- Animation script drafts (before validation and render)

### 2. Classification with Fallback

Vizion classifies prompts into templates, then validates extracted JSON. If confidence is low, it falls back or errors. The user sees a failure, not a broken video.

```python
if template_name == "unknown" or confidence < 0.7:
    return None, confidence  # fail visibly, do not guess
```

### 3. Read-Only Query Assistants

RAG over docs, design systems, or internal wikis. The model retrieves and summarizes. It does not write to prod databases.

### 4. Developer Productivity (Off the Critical Path)

Codegen in IDE, commit message suggestions, CI log summarization. If it is wrong, a human catches it before merge.

### 5. Content Transformation with Schema Validation

LLM outputs JSON matching a Zod/Pydantic schema. Invalid output rejects. Valid output passes through deterministic code.

```typescript
const schema = z.object({
  title: z.string().max(80),
  functions: z.array(z.object({
    expr: z.string(),
    label: z.string(),
    color: z.enum(['BLUE', 'RED', 'GREEN']),
  })),
});

const parsed = schema.safeParse(llmOutput);
if (!parsed.success) throw new ValidationError(parsed.error);
```

## Red Flags in PRDs and Pitch Decks

Push back when you see:

- "AI-powered access management"
- "Natural language billing adjustments"
- "Chatbot that can modify user permissions"
- "Auto-resolve security incidents" without human approval
- "LLM as orchestrator" for multi-step prod workflows with side effects

Ask: *If the model is wrong 1% of the time, what is the blast radius?* For auth at 10,000 users, 1% is 100 breaches. Unacceptable.

## A Simple Decision Framework

Before shipping AI in a feature, ask four questions:

1. **Can it cause irreversible harm?** (money, access, data loss)
2. **Do regulators or contracts require audit logs?**
3. **Can we write a deterministic test for the outcome?**
4. **Is there a human approval step before side effects?**

- **Yes to 1 or 2**: no AI execution. AI may draft or suggest only.
- **No to 3**: add schema validation and human review, or do not ship.
- **No to 4**: add an approval workflow before any write.

## Best Practices

1. **Draw a hard boundary**: model output is untrusted until validated
2. **Use enums and schemas**, not free-text, for anything that mutates state
3. **Log business events separately** from LLM inference logs
4. **Version models and prompts** so incidents are reproducible
5. **Fail visibly** when confidence is low. Silent guesses are worse than errors
6. **Keep kill switches** for AI features independent of core auth/billing paths

## Conclusion

AI in production is not a morality question. It is an architecture question. Auth, billing, permission changes, and audited actions need deterministic, testable code with clear actors and rollback paths. LLMs belong beside that code as assistants, classifiers, and drafters, not as replacements for it.

The teams that ship safely treat models like talented interns: great at first drafts, never allowed to push to main on auth or billing without review.

If you are adding AI to an enterprise product, map every proposed feature on the framework above. The uncomfortable "no" early saves the production incident later.

Happy shipping (the boring, reliable parts).
