Turning a plain-text idea into a Manim animation sounds simple until you try it in production. You describe `"Graph y = x² and show its shift to (x-2)²"`, an LLM writes Python, Manim renders a video, and the user sees the result. That is the promise behind [Vizion](https://github.com/leviackerman05/vizion) and its React frontend [vizion-ui](https://github.com/leviackerman05/vizion-ui).

In practice, the pipeline breaks in predictable places: prompts that look fine but classify wrong, generated code that passes syntax checks but calls hallucinated Manim APIs, and renders that fail because LaTeX is not installed. This article walks through how Vizion is structured, where each stage fails, and what we learned building a text-to-animation tool with Gemini and Manim CE.

## The Architecture: Two Repos, One Pipeline

Vizion splits cleanly across backend and frontend:

**Backend ([vizion](https://github.com/leviackerman05/vizion))**: Python + FastAPI
- `script_gen.py`: prompt to Manim Python script
- `renderer.py`: subprocess call to Manim CLI
- `prompt_engine/`: intent detection and prompt templates
- `template_engine.py`: structured template path for reliable layouts
- `main.py`: `/chat`, auth, and static video serving

**Frontend ([vizion-ui](https://github.com/leviackerman05/vizion-ui))**: React + Vite + TypeScript
- Chat panel for natural language input
- Video preview panel with generation states (`idle`, `generating`, `ready`, `error`)
- Supabase/Firebase auth on the UI side, JWT verification on the API
- shadcn/ui + Tailwind for the interface

The end-to-end flow:

```
User prompt → intent/template classification → code generation → validation → Manim render → MP4 URL
```

A single `/chat` request in `main.py` saves the user message, runs `generate_script()`, calls `render_manim_script()`, persists the assistant response with `video_url`, and returns the result to the UI.

## Stage 1: Prompt to Intent

Before any code is generated, Vizion needs to understand what kind of animation the user wants.

### Embedding-Based Intent Detection

`smart_intent_detector.py` uses `sentence-transformers` (`all-MiniLM-L6-v2`) to match prompts against intent descriptions:

```python
INTENT_DESCRIPTIONS = {
    "math_proof": "a mathematical identity or equation that needs visual proof",
    "graph_function": "a function that should be plotted using Axes and labeled curves",
    "concept_explanation": "a general abstract topic visualized using shapes or diagrams",
    "step_by_step_process": "an algorithm or logical process animated in sequence",
    # ...
}
```

The best cosine match picks the prompt template from `PROMPT_TEMPLATES` in `prompts.py`. If embeddings fail to load, the system falls back to keyword-based `intent_detector.py`.

**Failure mode**: vague prompts like `"Steps to make chicken curry"` map to `recipe_instruction` or `step_by_step_process`, but Manim is built for math visuals, not cooking tutorials. Intent detection returns a label; it does not guarantee the output will be good.

### Template Classification (The Reliable Path)

The newer `template_engine.py` takes a different approach. Instead of generating raw code immediately, it:

1. **Classifies** the prompt into a known template (`function_graph`, `algebraic_steps`, `geometric_proof`)
2. **Extracts structured JSON parameters** via Gemini (function expressions, proof steps, shape positions)
3. **Validates** parameters against template rules
4. **Generates code** from pre-built template classes with guaranteed layout

```python
def generate_from_template(user_prompt: str) -> Tuple[Optional[str], str]:
    template_name, confidence = classify_prompt(user_prompt)
    if not template_name:
        return None, "Could not classify prompt into a known template"

    params = extract_parameters(template_name, user_prompt)
    is_valid, error_msg = template_class.validate_params(params)
    code = template_class.generate_code(params)
    return code, "success"
```

If confidence is below `0.7` or extraction fails, `script_gen.py` falls back to raw LLM generation.

**Lesson**: templates beat raw generation for anything you ship repeatedly. Use the LLM to fill slots in code you already trust, not to invent layout from scratch every time.

## Stage 2: Prompt to Code

### Raw LLM Generation

When templates do not match, `generate_script_with_raw_llm()` sends the user prompt plus a intent-specific system prompt to Gemini (`gemini-2.0-flash` by default).

The system prompt in `prompts.py` is strict:

- Output **only** Python code, no markdown
- Exactly one class: `GeneratedScene(Scene)`
- Screen bounds, valid colors, allowed Manim objects listed explicitly
- Layout rules: `VGroup`, `buff=0.5`, titles at `.to_edge(UP)`

Despite all of this, models still hallucinate methods that do not exist in Manim CE 0.19.0.

### Code Extraction and Validation

`extract_code_from_response()` pulls code from markdown fences or finds `from manim import *` in the raw response.

Then multiple gates run before save:

```python
# Must contain a Scene class
if "Scene" not in script_code or "class" not in script_code:
    return ""

# Check for known hallucinated methods
invalids = check_for_invalid_manim_methods(script_code)

# Force class name for renderer compatibility
script_code = re.sub(r"class\s+\w+\s*\(", "class GeneratedScene(", script_code)

# Syntax check
ast.parse(script_code)
```

`script_validation.py` maintains a blocklist of invented methods like `add_coordinate_labels`, `highlight_segment`, and `move_arrow_to`. Warnings print but do not always block save. In hindsight, hard-blocking on any invalid method would have saved render time.

**Common failures at this stage**:

- **Empty response**: Gemini rate limit or bad API key. Fix: retry and show a clear error in the UI.
- **No `Scene` class**: model returned explanation text. Fix: stricter prompt, reject non-code responses.
- **Syntax error**: invalid Python from the model. Fix: `ast.parse()` gate before save.
- **Hallucinated Manim API**: model invents methods. Fix: blocklist plus template fallback.
- **Wrong LaTeX in MathTex**: unescaped backslashes. Fix: template path with validated params.

## Stage 3: Code to Render

`renderer.py` is deceptively short and deceptively fragile:

```python
def render_manim_script(script_path: str, class_name: str = "GeneratedScene") -> str:
    command = [
        "manim", "-pqh", script_path, class_name,
        "--media_dir", output_dir,
        "--output_file", "scene.mp4"
    ]
    subprocess.run(command, check=True)
    return f"/static/outputs/videos/{class_name}/480p15/scene.mp4"
```

**Failure modes in production**:

1. **Missing LaTeX**: `MathTex` requires `pdflatex` (MiKTeX or TeX Live). Manim fails with opaque errors if it is not installed.
2. **Subprocess timeout**: complex scenes hang the `/chat` endpoint. Vizion currently runs render synchronously inside the request.
3. **Class name mismatch**: CLI mode regex-extracts the class name; API path hardcodes `GeneratedScene`. Renaming only works when the regex path is used.
4. **Disk space**: `app/static/outputs/` grows with every generation. Git ignores it, but the server does not clean up.

The vizion-ui frontend handles this with explicit video states and cache-busting:

```typescript
const videoUrl = data.video_url + `?t=${Date.now()}`;
setVideoSrc(videoUrl);
setVideoState("ready");
```

On failure, users see `"Failed to generate video."` and `videoState` flips to `"error"`.

**Lesson**: never block an HTTP request on a full Manim render. Queue the job, return a job ID, poll or webhook when the MP4 is ready. We learned this the hard way waiting 30-90 seconds per `/chat` call.

## The Frontend Experience

[vizion-ui](https://github.com/leviackerman05/vizion-ui) wraps the pipeline in a chat-first UI:

- **Sidebar**: chat history loaded from `/chats`
- **ChatPanel**: message thread with filler responses while generating
- **VideoPanel**: preview of the rendered MP4

`Index.tsx` creates a new chat with a UUID on first message, calls `sendUserPrompt(userId, chatId, text)`, and maps the returned `video_url` to the preview panel.

Auth uses JWT tokens stored in `localStorage`, verified on the backend via `verify_token` before any chat operation. Signup/login go through Supabase on the API side.

The split between Firebase Auth (mentioned in the UI README) and Supabase (in the backend code) is worth unifying if you fork this project. Mixed auth layers create confusion during deployment.

## What We Would Do Differently

1. **Async render queue**: Celery, RQ, or a simple background task with status polling
2. **Template-first always**: expand `TEMPLATE_REGISTRY` before improving raw prompts
3. **Hard-fail on invalid Manim methods**: do not render code that fails validation
4. **Sandbox Manim execution**: run renders in isolated containers with LaTeX pre-installed
5. **Save intermediate artifacts**: store prompt, intent, raw LLM output, and script separately for debugging
6. **Timeout and cost limits**: cap prompt length and render duration per user

## Best Practices for Text-to-Code Pipelines

1. **Treat LLM output as untrusted input** until validated and sandboxed
2. **Constrain the output space**: templates, AST checks, allowlists for API calls
3. **Separate generation from execution**: code gen and Manim render should not share a request lifecycle
4. **Log every stage**: prompt, intent, template match, validation result, render exit code
5. **Show partial progress in the UI**: users wait minutes for video; filler messages help but job status helps more
6. **Version your prompts**: `BASE_SYSTEM_PROMPT` changed often; git tag what worked

## Conclusion

Vizion proves that text-to-animation is buildable with Gemini, FastAPI, and Manim. It also proves that the hard part is not the prompt. It is everything after the prompt: classification accuracy, code validation, render infrastructure, and honest error handling in the UI.

The template engine path (`classify → extract JSON → validate → generate`) is the most reliable piece of the system. The raw LLM path works for demos and fails for edge cases. If you are building something similar, start with templates for your top three use cases and treat open-ended generation as a fallback, not the default.

Clone the repos, run a prompt through CLI mode (`python -m app.main`), and watch where it breaks. That is faster than any tutorial.
